Bitchat lets two phones exchange encrypted messages over Bluetooth LE with no internet connection, no server, and no phone number. If a mesh can carry a message, it can carry a transaction. This fork adds a full Monero wallet to Bitchat Android — send, receive, and search transactions over the same offline mesh the app already uses for chat.

The Premise

If a Mesh Can Carry a Message, It Can Carry a Transaction

Bitchat is a decentralised, peer-to-peer messaging app that moves encrypted messages over Bluetooth LE mesh networks — no internet required, no central server, no account tied to a phone number. Devices discover each other directly, relay packets for one another over multiple hops, and store-and-forward for peers who are temporarily offline. It is, in effect, a working proof that a phone-to-phone mesh can move meaningful data without any infrastructure at all.

Monero is a privacy-preserving cryptocurrency built around the same instinct applied to money: transactions that are untraceable and unlinkable by default, with no bank, no card network, and no intermediary who can freeze or inspect a payment. The two projects solve adjacent problems — one moves messages without a server, the other moves value without a custodian — and neither depends on the public internet in the way a conventional app does.

Since messages can be sent, then so can crypto. All it takes is a switch that determines what kind of message is being sent — or it can automatically identify the kind of information being carried.

— Working premise behind this fork

This fork of bitchat-android, on the with-monero branch, follows that premise through: it wires a real Monero wallet — built on the same Monerujo xmrwallet JNI bindings (com.m2049r.xmrwallet) used by that well-known standalone Monero wallet app — into Bitchat's existing Bluetooth mesh, encryption layer, and chat UI, so that a payment is just another message type flowing through a channel that already exists.

💭
A decade-old idea, revisited

I had a version of this idea a long time before this fork existed. Back in December 2016 I pitched a Bluetooth-based chat app I called Toastr (originally "Toaster"), in a written proposal at the time — proximity-based messaging between strangers in the same physical space, over Bluetooth, no server involved. What's below is not that pitch — it's someone else's mesh, extended with a Monero wallet — but the shape of the idea, phone to phone with no infrastructure in between, is the same one I wrote down nearly a decade earlier.

The Extension

A Monero Wallet Grafted Onto Bitchat's Existing Mesh

The integration lands as a self-contained com.bitchat.android.monero package alongside the Monerujo xmrwallet wallet library (the JNI bindings behind the well-known Monerujo Android wallet), plus native headers for Wallet, WalletManager, and PendingTransaction. Nothing about Bitchat's core mesh, encryption, or UI framework was replaced — the wallet code plugs into it as an additional capability that rides on the same Bluetooth transport and the same ChatViewModel that already renders every other message in the app.

The footprint is substantial for a fork: roughly 5,100 lines across twelve new files — a wallet state machine, a transaction manager with multi-node failover, a message protocol for payments and payment requests, chat-bubble UI components for sent and received transactions, and a Bluetooth transfer layer that fragments transaction data to fit Bitchat's existing packet size limits.

Technical Deep Dive

Four Pieces, Wired Into One Wallet

👛
Component 01

WalletSuite

The core wallet engine — roughly 1,800 lines wrapping the Monero JNI library. Manages wallet lifecycle, sync, rescan, and transaction creation through an explicit state machine (IDLE, SYNCING, RESCANNING, CLOSING, TRANSACTION) so that overlapping operations can't corrupt wallet state.

🌐
Component 02

MoneroTransactionManager

Queries one or more Monero daemon nodes in parallel to look up a transaction by ID — useful because a transaction can propagate to some nodes before others. Node credentials are encapsulated behind getters, and connections can be forced over TLS.

📡
Component 03

MoneroChatTransferManager

Sits between the wallet and the chat UI. Supports two ways to hand a peer money: a lightweight TXID_BASED flow (broadcast the transaction, share the ID) and a heavier BLOB_BASED flow that transfers the raw signed transaction blob over the mesh itself.

💬
Component 04

MoneroMessageHandler

Defines the payment protocol carried inside ordinary Bitchat messages — payments, payment requests, address shares, status updates, and transaction search — as immutable JSON-typed messages, with a legacy bracket-prefixed format ([MONERO_PAYMENT], [MONERO_TXID], etc.) kept for compatibility.

End-to-End Flow

Sending XMR to Someone Standing Near You

Because Bitchat has no concept of a persistent internet connection, a Monero send has to work whether or not either device is online in the conventional sense — only the mesh link to the peer, and eventually a connection to a Monero daemon to broadcast the transaction, are required.

Monero Send — TxID-Based Flow (default)
01
Peer selected, address exchanged

The sender picks a private peer from the existing Bitchat peer list. If the receiver's Monero address isn't already known, it's requested and shared as a normal mesh message — no separate contact system.

✓ Reuses Bitchat's peer discovery
02
Transaction created and signed locally

WalletSuite builds and signs the transaction on-device through the wallet's JNI layer, moving into a TRANSACTION state that blocks other wallet operations until it completes.

✓ Signing never leaves the device
03
Transaction broadcast, TxID shared over the mesh

The signed transaction is submitted to a configured Monero node. Its transaction ID is then sent to the recipient as an ordinary encrypted Bitchat message, rendered as a payment bubble in the chat thread rather than plain text.

✓ Chat bubble UI for sent/received XMR
04
Recipient confirms independently

The recipient's app can look the TxID up across multiple configured nodes via MoneroTransactionManager, rather than trusting the sender's word for it — useful when the transaction hasn't propagated to every node yet.

✓ Multi-node lookup, first result wins

This is the actual routing logic that decides which of the two flows a send takes:

MoneroChatTransferManager.kt — Flow Mode Selection Kotlin
enum class TransactionFlowMode { TXID_BASED, BLOB_BASED, BOTH } fun handleMoneroSend( amount: String, selectedPrivatePeer: String?, canReceiveMonero: Boolean, receiverMoneroAddress: String?, onSuccess: () -> Unit, onError: (String) -> Unit ) { if (selectedPrivatePeer == null) { onError("No peer selected"); return } if (!canReceiveMonero) { onError("Peer cannot receive Monero"); return } when (flowMode) { TXID_BASED -> sendWithTxIdFlow(amount, selectedPrivatePeer, receiverMoneroAddress, onSuccess, onError) BLOB_BASED -> sendWithBlobFlow(amount, selectedPrivatePeer, receiverMoneroAddress, onSuccess, onError) BOTH -> sendWithTxIdFlow(amount, selectedPrivatePeer, receiverMoneroAddress, onSuccess) { error -> sendWithBlobFlow(amount, selectedPrivatePeer, receiverMoneroAddress, onSuccess, onError) // fallback } } }

On the wallet side, the state machine that guards every sensitive operation is deliberately narrow — a transaction can only be created or submitted from IDLE, and every transition back to IDLE happens in a finally block so a failed send can't leave the wallet stuck mid-operation. Here's sendWithTxIdFlow() itself, the code that actually runs when TXID_BASED is selected:

MoneroChatTransferManager.kt — sendWithTxIdFlow() Kotlin
val cachedBalance = walletSuite.getBalanceValue() val cachedUnlockedBalance = walletSuite.getUnlockedBalanceValue() walletSuite.sendTransaction(receiverMoneroAddress, amountXmr, cachedBalance, cachedUnlockedBalance, object : WalletSuite.TransactionCallback { override fun onSuccess(txId: String, amount: Long) { val msg = messageHandler.createTransactionIdMessage( txId = txId, amount = WalletSuite.convertAtomicToXmr(amount), toAddress = receiverMoneroAddress ) sendMessageCallback(selectedPrivatePeer, msg) // rides out as a normal bitchat DM onSuccess() } override fun onError(error: String) = onError(error) })

A second, heavier path — BLOB_BASED — exists for moving the raw signed transaction blob itself across the mesh, rather than relying on the sender's own device to reach a daemon at all. createTxBlob() builds and signs the transaction exactly like the TxID flow, but stops one step short: it never calls submitTransaction(). Instead it writes the signed transaction to a temp file and reads the raw bytes back out as base64 — a self-contained artifact that can sit on a phone with no signal indefinitely:

WalletSuite.java — createTxBlob() (offline signing, no broadcast) Java
public void createTxBlob(String to, String amount, TxBlobCallback cb) { executorService.execute(() -> { long atomic = Helper.getAmountFromString(amount); PendingTransaction pendingTx = wallet.createTransaction( to, "", atomic, 15, // mixin / ring size PendingTransaction.Priority.Priority_Default.getValue(), 0); String txId = pendingTx.getFirstTxId(); File tempFile = new File(context.getCacheDir(), txId + ".tx"); pendingTx.commit(tempFile.getAbsolutePath(), true); // signs, does NOT broadcast byte[] raw = Files.readAllBytes(tempFile.toPath()); String b64 = Base64.encodeToString(raw, Base64.NO_WRAP); cb.onSuccess(txId, b64); // this blob is what travels over the mesh }); }

A blob under 100 KB rides out as an ordinary chat message prefixed [XMR_TX_BLOB]; a larger one falls back to Bitchat's own file-transfer path. Whoever ends up with connectivity — sender, receiver, or a relaying peer — closes the loop by submitting the raw bytes to a daemon. That's what happens automatically the moment a blob arrives on the receiving end:

WalletSuite.java — submitTxBlob() (broadcast on receipt) Java
public void submitTxBlob(byte[] blobBytes, TxBlobCallback callback) { executorService.execute(() -> { StringBuilder hexBuilder = new StringBuilder(blobBytes.length * 2); for (byte b : blobBytes) { hexBuilder.append(String.format("%02x", b & 0xff)); } String txId = wallet.submitTransaction(hexBuilder.toString()); wallet.store(); updateBalancesFromWallet(); callback.onSuccess(txId, Base64.encodeToString(blobBytes, Base64.NO_WRAP)); }); }

In the current branch, the mesh-integration hooks that would fully automate multi-hop relaying of very large blobs are still stubs — the signing and submission code above is real and works, but the app defaults to TXID_BASED for day-to-day sending, which is fully wired end to end.

Under the Hood

A Payment Is Just Another Message Type

The cleanest expression of the whole idea is in how a Monero payment travels through the same channel as a text message: it is serialized as a small typed JSON object, indistinguishable in transport terms from any other Bitchat packet, and only interpreted specially by the UI layer once received.

That extends to verification, too. A transaction broadcast from a phone with a weak or intermittent connection may reach one public node before it reaches another, so MoneroTransactionManager holds a list of daemon connections rather than just one, and queries all of them in parallel when a peer needs to confirm a payment actually landed:

This is what powers the transaction-search flow in the UI — paired chat-protocol messages ([MONERO_SEARCH_TX] / [MONERO_TX_FOUND] / [MONERO_TX_NOT_FOUND]) let one peer explicitly ask another "did this transaction land for you?", which matters precisely because on a mesh, "confirmed for me" and "confirmed for you" aren't guaranteed to be the same moment.

Honest Assessment

What's Solid, What's Still Debt

This is a fork with a real, working core and some clearly-labelled unfinished edges — worth stating plainly rather than glossing over, since the project's own code review notes are admirably specific about it.

🔧
Two GATT servers is the one to watch

The most consequential remaining issue is architectural: a dedicated MoneroBluetoothService reimplements its own BLE GATT server and client rather than extending Bitchat's existing BluetoothGattServerManager. Running two GATT servers on the same device risks resource conflicts on most Android hardware. What already works from this class — Monero-capability advertisement during peer discovery and the Java/Kotlin interop bridge — is useful; the parallel GATT stack is the part that needs a structural refactor before this should be considered production-ready.

Conclusion

A Mesh That Was Always Capable of More Than Chat

What makes this fork interesting isn't novelty in either half — Bluetooth mesh messaging and Monero wallets both exist independently and are well understood. It's the fact that grafting one onto the other required surprisingly little new infrastructure: the mesh, the encryption, the peer discovery, and the chat UI were all already there, built for text. A payment is, structurally, just a message with a different payload.

That's the same intuition I had almost a decade before this repository existed, in a much simpler form — proximity-based Bluetooth messaging as a platform for more than chat. Seeing it actually built, with a real wallet library and a real mesh underneath it, is a good reminder that the interesting ideas are rarely the hard part. The execution — state machines that don't corrupt wallets, GATT servers that don't collide, transaction flows that degrade gracefully when a peer goes offline — is where the real work lives, and this fork is refreshingly candid about exactly how much of that work remains.

A mesh built to carry a sentence can be taught to carry a transaction. The hard part was never the wire — it was making sure the wallet on the other end doesn't fall over.
